Introduction & Scope

CivicPulse is a production-grade end-to-end machine learning system that ingests public civic complaint records, engineers temporal and NLP features, trains a multi-algorithm ML suite, and surfaces predictions as an interactive ward-level risk heatmap. The system answers two questions simultaneously:

Regression Task
How long will this complaint take to resolve?
Predicts resolution days from filed date using regression + gradient boosting
Classification Task
Will this complaint never be resolved?
Binary classifier for complaints censored at 90 days — the "black hole" detector

Problem Statement

India's Smart Cities Mission and state civic portals receive millions of complaints annually across categories like garbage collection, road damage, water supply disruptions, and electricity faults. Despite digitisation, two structural problems persist:

Pain Point 1 — Zero Citizen Visibility Citizens who file complaints on platforms like Swachh Bharat Mission portals, MyGov, or state 311-equivalents receive no estimated resolution timeline. They have no way to know if their complaint has been deprioritised, lost in a departmental queue, or is actively being worked on.
Pain Point 2 — Ward Administrator Blind Spots Ward-level administrators cannot identify systemic resolution failures — specific ward × complaint-type combinations that chronically go unresolved. Without this visibility, resource allocation remains reactive rather than predictive.

CivicPulse addresses both: it predicts time-to-resolution for new complaints, and identifies "black hole" patterns — ward-level clusters where complaints are disproportionately likely to never close within 90 days.

Socrata API Quick-Start

# Pull 500K recent NYC 311 records via Socrata SODA API (no key required)
# Endpoint: https://data.cityofnewyork.us/resource/erm2-nwe9.json

import requests, pandas as pd

BASE_URL = "https://data.cityofnewyork.us/resource/erm2-nwe9.json"

def fetch_nyc311(limit=500_000, offset=0):
    params = {
        "$limit": limit,
        "$offset": offset,
        "$order": "created_date DESC",
        "$where": "created_date > '2020-01-01'",
        "$select": "unique_key,created_date,closed_date,complaint_type,descriptor,agency,borough,community_board,latitude,longitude,status,resolution_description"
    }
    r = requests.get(BASE_URL, params=params, timeout=120)
    return pd.DataFrame(r.json())

df = fetch_nyc311()
df.to_csv("nyc311_raw.csv", index=False)
Spark Ingestion Tip For large-scale ingestion via PySpark on Databricks, use the Socrata API paginated loop and write directly to S3 Bronze as Delta. Set batch size to 100K rows, paginate with $offset, and run in a Databricks Job with retries on HTTP 429.

NLP Text Pipeline

The descriptor column in NYC 311 complaints and equivalent text fields in Indian data contain unstructured free text — "Broken manhole cover on the road near the bus stop," "Garbage not collected for 3 days," etc. CivicPulse uses a 5-stage NLP pipeline to extract signal from this text.

Why NLP Makes This Unique Most civic data ML projects treat complaint type as a simple categorical variable. The free-text description contains urgency signals ("emergency," "flooding," "dangerous"), specificity signals (length, number of specific nouns), and topic clusters that predict resolution speed independently of the complaint category label.
01
Text Cleaning & Normalisation
Lowercase, remove special characters, strip HTML entities, expand contractions. Stored in Silver as clean_description. Library: PySpark regexp_replace + Python re.
02
Urgency Keyword Scoring
Rule-based scoring: presence of terms like "emergency," "danger," "flooding," "leaking," "no water 3 days," "pothole accident" maps to a float urgency_score (0–1). These keywords are higher correlated with faster resolution (the squeaky wheel) AND with never-resolved (systemic failures ignored). Feature: urgency_score.
03
TF-IDF Vectorisation + Truncated SVD
Scikit-learn TfidfVectorizer (max_features=5000, ngram_range=(1,2)) → TruncatedSVD (n_components=50). Produces 50 dense numerical features per complaint capturing vocabulary similarity across complaint descriptions. Stored as tfidf_feat_1..50 in Gold.
04
LDA Topic Modelling
Latent Dirichlet Allocation (sklearn LatentDirichletAllocation, n_topics=12) discovers latent complaint themes: e.g., Topic 0 = "water leakage pipe burst," Topic 4 = "noise construction night," Topic 9 = "road pothole accident risk." Each complaint gets a topic_id (dominant topic) — this is a powerful categorical feature for the downstream XGBoost.
05
NLP-Only Classifier (Standalone Model)
A separate LightGBM or Logistic Regression trained only on TF-IDF features to predict complaint_category from the raw description text — this validates that the text alone carries category signal, and generates nlp_predicted_category as a cross-validation feature for the main models. Target: >80% top-3 accuracy.

NLP PySpark Implementation Sketch

from pyspark.ml.feature import Tokenizer, StopWordsRemover, HashingTF, IDF
from pyspark.ml import Pipeline
from pyspark.sql.functions import col, udf, length
from pyspark.sql.types import DoubleType
import re

# Urgency keyword scoring UDF
URGENCY_KEYWORDS = ["emergency", "flood", "danger", "leak", "no water", "accident", "broken"]

def compute_urgency(text):
    if not text: return 0.0
    hits = sum(1 for kw in URGENCY_KEYWORDS if kw in text.lower())
    return min(hits / 3.0, 1.0)

urgency_udf = udf(compute_urgency, DoubleType())

silver_df = silver_df.withColumn("urgency_score", urgency_udf(col("clean_description")))
silver_df = silver_df.withColumn("description_length", length(col("clean_description")))

# TF-IDF pipeline in Spark MLlib
tokenizer = Tokenizer(inputCol="clean_description", outputCol="words")
remover = StopWordsRemover(inputCol="words", outputCol="filtered_words")
hashing_tf = HashingTF(inputCol="filtered_words", outputCol="raw_features", numFeatures=5000)
idf = IDF(inputCol="raw_features", outputCol="tfidf_features")

nlp_pipeline = Pipeline(stages=[tokenizer, remover, hashing_tf, idf])
model = nlp_pipeline.fit(silver_df)
gold_df = model.transform(silver_df)

Delta Live Tables Pipeline DAG

## Notebook: 02_clean_silver.py
import dlt
from pyspark.sql.functions import col, datediff, when, to_timestamp, dayofweek, hour, month

@dlt.table(
    name="complaints_clean",
    comment="Silver layer: cleaned and labelled complaint records",
    table_properties={"quality": "silver"}
)
@dlt.expect_or_drop("valid_complaint_id", "complaint_id IS NOT NULL")
@dlt.expect_or_drop("valid_created_date", "created_date IS NOT NULL")
def complaints_clean():
    bronze = dlt.read_stream("complaints_raw")
    return (bronze
        .withColumn("resolution_days",
            datediff(col("closed_date"), col("created_date")))
        .withColumn("never_resolved",
            when((col("closed_date").isNull()) &
                 (datediff("current_date", col("created_date")) > 90), 1).otherwise(0))
        .withColumn("dow_filed", dayofweek(col("created_date")))
        .withColumn("hour_filed", hour(col("created_date")))
        .withColumn("month_filed", month(col("created_date")))
        .filter(col("complaint_id").isNotNull())
    )

Historical Aggregation PySpark Pattern

from pyspark.sql import Window
from pyspark.sql.functions import count, avg, when, col

# Ward-level black hole rate (trailing 12 months)
ward_window = Window.partitionBy("ward_id")

ward_agg = (silver_df
    .filter(col("created_date") >= add_months("current_date", -12))
    .groupBy("ward_id", "complaint_category")
    .agg(
        avg(when(col("never_resolved") == 1, 1).otherwise(0)).alias("ward_blackhole_rate"),
        avg("resolution_days").alias("ward_avg_resolution_days"),
        count("*").alias("ward_cat_complaint_count")
    )
)

# Join back to silver
gold_df = silver_df.join(ward_agg, on=["ward_id", "complaint_category"], how="left")

ML Model Suite — 4 Algorithms

Design Decision — Why Multiple Models? Single-algorithm projects are weak portfolio pieces. CivicPulse runs 4 algorithms across 2 tasks (regression + classification), compares them in MLflow, and selects the best per task for the production heatmap. This demonstrates model selection discipline, not just implementation.

Task 1 — Regression: Predict Resolution Days

Regression · Algorithm 1
XGBoost Regressor
Gradient boosted trees. Handles non-linear feature interactions (ward × dept × complaint_type). Primary production model for regression task. Supports feature importance output.
n_estimators=500 · max_depth=7 · learning_rate=0.05 · subsample=0.8
Regression · Algorithm 2
Random Forest Regressor
Ensemble of 300 decision trees. More stable than XGBoost on smaller training sets; useful baseline to diagnose overfitting. Lower variance at expense of slight bias.
n_estimators=300 · max_depth=15 · min_samples_leaf=5

Task 2 — Classification: Predict "Never Resolved" (Black Hole)

Classification · Algorithm 3
XGBoost Classifier
Primary black hole detector. Binary output: 1 = likely never resolved. Use scale_pos_weight to handle class imbalance (black holes are ~15–20% of complaints). Calibrate probabilities with Platt scaling for reliable ward-level risk scores.
scale_pos_weight=4 · eval_metric=auc · threshold=0.40
Classification · Algorithm 4
Logistic Regression (L2)
Interpretable baseline. Produces well-calibrated probabilities directly. Essential for stakeholder explainability — ward administrators can see which features most strongly drive the "never resolved" prediction (department load, ward history, complaint type).
C=0.1 (L2) · class_weight=balanced · max_iter=500

Bonus — NLP Classifier (Text-Only Baseline)

NLP · Algorithm 5 (Bonus)
LightGBM on TF-IDF Features
Trained only on the 50 TF-IDF SVD features + urgency_score + description_length. Predicts whether the complaint will be resolved within SLA. Serves as a text-signal ablation study — quantifies how much the description text contributes to prediction beyond the structured features alone. Target: >72% accuracy independently.
LightGBM · n_estimators=200 · features: TF-IDF only (no structured features)

Production Build Steps

Environment Databricks Community Edition (free) is sufficient to build and demo this. Use a 14.3 LTS ML Runtime cluster with a single-node config (15.3 GB RAM). AWS S3 bucket (free tier) for data storage.

Phase 1 — Infrastructure Setup

Step 1.1 — Create Databricks workspace (Community Edition or AWS trial). Enable Unity Catalog. Create catalog: civicpulse, schemas: bronze, silver, ml, output.
Step 1.2 — Create S3 bucket: s3://civicpulse-data/. Folders: raw/nyc311/, raw/india/, raw/shapefiles/.
Step 1.3 — Configure Databricks ↔ S3 IAM role with s3:GetObject, s3:PutObject, s3:ListBucket. Set as instance profile on cluster.
Step 1.4 — Install cluster libraries: xgboost==2.0.3, lightgbm, folium, keplergl, scikit-learn, nltk, geopandas, shap.

Phase 2 — Data Ingestion

Step 2.1 — Download NYC 311 data via Socrata API (use pagination script from Section 4). Target: 5M–10M records (2020–2024). Upload CSVs to s3://civicpulse-data/raw/nyc311/.
Step 2.2 — Download India data from data.gov.in (bulk CSV). Upload to s3://civicpulse-data/raw/india/. Note: India data is smaller (100K–500K rows); use for validation only.
Step 2.3 — Download ward shapefiles. NYC: geo_export.geojson from NYC Open Data. India: from data.gov.in ward boundaries. Upload to s3://civicpulse-data/raw/shapefiles/.
Step 2.4 — Configure Auto Loader in notebook 01_ingest_bronze to watch the S3 raw drop zones. Set cloudFiles.format = "csv" with schema inference + schema evolution mode.

Phase 3 — Pipeline Development

Step 3.1 — Build and test DLT pipeline 02_clean_silver. Verify: null handling, duration computation, never_resolved label logic. Run DESCRIBE TABLE civicpulse.silver.complaints_clean to verify schema.
Step 3.2 — Build aggregation notebook 03_aggregate_silver. Test ward_blackhole_rate by filtering known black-hole wards (complaints open >90 days). Spot-check against raw data.
Step 3.3 — Build NLP pipeline 04_nlp_features. Verify TF-IDF shape: (N_rows, 50). Spot-check topic_id: print top-10 words per LDA topic; they should form coherent themes.
Step 3.4 — Build Gold materialised view 05_build_gold. Final count check: Gold rows should match Silver (no accidental fan-out from JOINs). Assert: SELECT COUNT(*) FROM civicpulse.ml.training_complaint_states.

Phase 4 — Model Training & Evaluation

Step 4.1 — Train/test split: 80/20, stratified by complaint_category for classification task. For regression, use time-based split (train on 2020–2023, test on 2024).
Step 4.2 — Train all 4 models in 06_train_models. Log each run to MLflow with full params + metrics + feature importance artifacts. Use mlflow.autolog() for XGBoost.
Step 4.3 — Generate SHAP explanations for XGBoost models. Log shap_summary_plot.png as MLflow artifact. This is your key explainability asset for interviews.
Step 4.4 — Register the best model per task in Unity Catalog: civicpulse.ml.resolution_regressor and civicpulse.ml.blackhole_classifier. Set alias @champion.

Phase 5 — Scoring & Visualisation

Step 5.1 — Run batch scoring in 07_score_wards: load champion models from Unity Catalog, score all currently open complaints, aggregate predictions to ward level, write to civicpulse.output.ward_risk_scores.
Step 5.2 — Build Folium heatmap in 08_visualise: load ward risk scores, merge with shapefile geometries, create choropleth coloured by blackhole_risk_score, add popup tooltips showing open count + predicted days + risk tier.
Step 5.3 — Export heatmap as standalone ward_heatmap.html. Download and host on GitHub Pages or include in README as screenshot. This is your demo asset.
Step 5.4 — Schedule the full pipeline as a Databricks Job (Workflow): Trigger daily at 2:00 AM. Stages: Ingest → Clean → Aggregate → Score → Visualise.

Risk Register

HIGH India dataset sparsity — data.gov.in complaint datasets are often small (10K–50K rows) and lack closed_date, making them poor training data.
Mitigation: Use NYC 311 as primary training; India as validation only.
HIGH Class imbalance — never_resolved class (black holes) is only ~15–20% of complaints. Naive model predicts 0 always.
Mitigation: scale_pos_weight in XGBoost; oversample with SMOTE on train set.
MEDIUM Geographic transferability — NYC 311 model trained on US complaint culture; Indian ward patterns differ significantly.
Mitigation: Fine-tune on Indian data subset; treat as domain adaptation problem.
MEDIUM Databricks Unity Catalog S3 IAM permission errors (s3:PutObject missing) blocking model registration.
Mitigation: Workaround: load via mlflow.xgboost.load_model from run URI; fix IAM separately.
MEDIUM API rate limiting on Socrata — free tier has limits on concurrent requests.
Mitigation: Add time.sleep(0.5) between pagination calls; use exponential backoff.
LOW TF-IDF feature explosion — 5000 features without dimensionality reduction can cause OOM.
Mitigation: TruncatedSVD to 50 components already planned. Monitor cluster memory.
